🏛️ 软件架构师审计

审计范围：架构分层、状态管理、工作流引擎、类型系统、API抽象、反模式、可扩展性、可维护性、ADR

关键发现

P0 表达式求值存在代码注入漏洞 helpers.ts:31-39
evaluateExpression 使用 with(payload) { return ${expression}; } 直接执行用户输入的字符串。with() 虽限制变量查找范围，但攻击者仍可通过原型链污染或全局对象访问执行任意代码。
建议：短期使用 jexl 或 jsonata 等安全表达式解析器；中期实现 AST 解释器；必须对表达式进行语法白名单校验。
P0 God Store 导致不必要的重渲染 stores/index.ts
单一 AppStore 包含所有状态，任何组件订阅任一字段变化都可能触发重渲染。chatSlice 已达 684 行，同时处理消息 CRUD、流式响应、窗口管理、Agent 转发。
建议：按领域拆分为独立 Zustand store（useChatStore、useWorkflowStore），使用 useShallow 优化订阅。
P0 WorkflowNodeData 类型完全开放 types/models.ts:126-129
WorkflowNodeDataBase & Partial<...> & Record<string, unknown> 使得任意字段都可合法存在，编译器无法捕获类型错误。
建议：改用 Discriminated Union：type WorkflowNodeData = ({type:'agent'} & AgentData) | ({type:'code'} & CodeData) | ...
P0 缺少架构决策记录 (ADR) docs/ 目录
项目中无任何 ADR 文档。Store 拆分、BFS 引擎、三级模型链等关键决策的"为什么"未记录，新开发者难以理解设计意图。
建议：创建 docs/adr/ 目录，记录 Zustand 选择、BFS 串行模型、三级解析链等决策。
P1 Slice 间隐式耦合 baseSlice.ts:259-341
fetchInitialData 直接设置其他 slice 的状态（settings、sessions、workflows），破坏边界独立性。所有 slice 的 set: any, get: any 完全绕过 TypeScript 类型检查。
建议：引入泛型约束 StoreSet / StoreGet；fetchInitialData 拆分为各 slice 的独立初始化 action。
P1 废弃 API 仍在被调用 api.ts:28, PromptGenerator.tsx
fetchFromModel 已标记 @deprecated 但 PromptGenerator.tsx 仍直接调用，维护两套 provider 解析逻辑。
建议：统一使用 fetchFromResolvedConfig，移除废弃函数。
P2 无代码分割/懒加载 App.tsx
所有视图组件同步导入，未使用 React.lazy。首次加载包含全部工作流画布、设置页等代码。
建议：视图级代码分割：const WorkflowCanvas = React.lazy(() => import(...))

⚡ 快速原型师审计

审计范围：技术栈选择、开发速度、MVP完整性、模块化、预构建组件、反馈收集、生产过渡、技术债务

关键发现

P0 P0 验证缺口阻塞发布 TODO 9.1 / 9.2
6+ 项浏览器验证全部未完成（single chat 完整链路、workflow chat、窗口交互、转发链路、状态隔离、对比度/点击区）。功能正确性未确认即准备发布。
建议：冻结新功能开发，集中 1 周完成所有 P0 验证清单。引入 Playwright E2E 测试覆盖核心用户旅程。
P0 子工作流节点是空壳实现 subWorkflowExecutor.ts
subWorkflowExecutor 仅返回 payload，无实际递归执行逻辑。P1 功能不可用，影响工作流引擎的完整性。
建议：调用 executeWorkflow 实现递归执行，传递参数和回传结果。预计 1-2 天工作量。
P1 Rust 后端严重欠利用 src-tauri/src/
Rust 层仅包含基础 CRUD 和表定义（~420 行），沦为"数据库胶水层"。未发挥 Tauri 在系统级集成、性能、安全方面的优势。
建议：明确 Rust 后端职责边界；考虑将表达式求值、API Key 加密、文件系统监控等移至 Rust 层。
P1 生产化准备严重不足 全局
无代码签名（Windows/macOS 会拦截未签名应用）、无自动更新器（Tauri Updater）、无落地页、无 GitHub Releases 流程。
建议：申请代码签名证书；集成 Tauri Updater；创建最小化落地页；建立 GitHub Actions 发布流水线。
P1 零用户反馈收集机制 全局缺失
无错误上报（Sentry）、无使用分析（Mixpanel）、无应用内反馈入口。产品迭代完全依赖主观判断。
建议：集成 Sentry 最小化错误上报；扩展"一键导出诊断包"功能；在设置页添加反馈入口。
P2 首次启动体验完全缺失 全局
新用户打开应用面对空列表，无引导、无示例高亮，流失风险高。
建议：首次启动时自动加载 1-2 个示例工作流和 Agent，避免空状态。

开源项目无 LICENSE 文件，存在法律风险，也会阻碍潜在贡献者参与。

P0 CONTRIBUTING.md 缺失 根目录
无贡献指南、无 Issue/PR 模板、无分支命名规范。外部开发者不知如何参与。

整个 src/ 目录仅 17 处 JSDoc 注释。大量核心函数（fetchFromProvider、runAgentResponse、createBaseSlice 等）无注释。

P1 文档同步严重滞后 docs/ 目录
多处文档状态与代码不符：plan-dynamic-agent.md 写"Plan 阶段未编码"但代码已实现；Features.md 写"设计中"但 Dynamic Agent 已上线；Development.md 版本标记为 v0.4.0 但代码已 v0.4.1+。

P1 README 缺少关键要素 README.md
无界面截图/GIF、Tauri 环境安装指引缺失、Node 版本未锁定（无 .nvmrc）、无徽章。

P2 文档入口分散 docs/ 目录
README.md、CLAUDE.md、Development.md 都可作为入口，新读者不知从何开始。DESIGN.md 与 Design_Specs.md 命名容易混淆。
建议：重命名 DESIGN.md 为外部参考文档；在 CLAUDE.md 中添加阅读路径推荐。

🔄 自主优化架构师审计

审计范围：多供应商路由、熔断器、成本管控、重试策略、超时控制、错误处理、影子测试、监控告警、并发控制、安全护栏

关键发现

P0 maxTokens/temperature 未传入 streamText api.ts:103
fetchFromProvider() 调用 streamText({ model, system, prompt }) 时未传入 maxTokens 和 temperature。用户设置被完全忽略，可能导致意外高额账单。
建议：立即修复：streamText({ ..., maxTokens: config.maxTokens, temperature: config.temperature })
P0 安全护栏多重缺陷 api.ts, httpExecutor.ts
API Key 明文存储在 JSON 和 SQLite 中；无输入长度限制；无 prompt injection 检测；localStorage fallback 泄露敏感数据。
注：HTTP 节点可访问内网地址为产品设计要求，不作为缺陷。
建议：使用 Tauri Keychain/Stronghold 存储 API Key；禁用含 API Key 的 localStorage 回退；增加输入长度限制。
P1 完全缺失熔断器机制 全局缺失
代码中无任何 circuit breaker、failure counter、half-open 状态。当供应商持续故障（429/500/503）时，系统会持续发送请求。
建议：为每个 provider 引入内存级熔断器：记录连续失败次数，超过阈值后进入 open 状态（冷却期 30s），half-open 后放行探测请求。
P1 聊天层无重试、无超时 chatSlice.ts:54
runAgentResponse() 无任何重试逻辑和超时控制。API 调用失败直接报错，流式响应可能无限挂起。工作流层的重试策略与聊天层不统一。
建议：在 runAgentResponse() 中引入 withTimeout（默认 60s）和基于错误类型的重试（仅重试 429/503/timeout/network error）。
P1 并发调用无限制 chatSlice.ts:414
sendMessageToAgents() 使用 Promise.all(agents.map(...)) 无任何并发限制。10+ Agent 同时触发会发起 10+ 个 LLM 请求，冲垮供应商 rate limit。
建议：引入 p-limit 或 Semaphore，限制同时 LLM 调用数为 3-5 个。实现 ExecutionOptions.concurrency 的实际逻辑。
P1 无智能路由与降级 agentProviderRouter.ts
三级解析链是静态配置，无基于延迟/成本/可用性的动态选择。注释明确写明 "No fallback — any missing/invalid config throws immediately"。
建议：增加跨供应商 fallback 链：首选失败后按优先级尝试其他 isEnabled 供应商。为每个 provider 增加健康检查评分。
P2 无影子测试 / A/B 测试 全局缺失
无法在不干扰用户的情况下测试新模型。引入新供应商需直接切换生产流量。
建议：增加 shadowProviderId/shadowModelId 字段，后台并行发送到影子模型仅记录结果。

📊 数据工程师审计

审计范围：Schema设计、持久化策略、数据迁移、数据质量、一致性、可观测性、备份恢复、配置管理、安全、可扩展性

关键发现

P0 数据库零索引 db.rs:40-114
所有表零索引。chat_messages 按 session_id 查询和 chat_sessions 按 workspace_id 查询将全表扫描，消息量增长后性能急剧恶化。
建议：立即添加：CREATE INDEX idx_chat_messages_session ON chat_messages(session_id) 等至少 3 个高频索引。
P0 Agent 双写不一致 baseSlice.ts:213-246
addAgent 同时写 SQLite 和 JSON，但 updateAgent/batchUpdateAgents 只写 JSON 不写 SQLite。重启后 fetchInitialData 优先读 JSON，SQLite 数据永远停留在旧版本。
建议：以 SQLite 为唯一持久化层，JSON 仅用于导出/迁移。补全 Rust update_agent/delete_agent 命令。
P0 相对路径导致数据丢失风险 db.rs:41, db.rs:419
Connection::open("simperstudio.db") 和 PathBuf::from("config") 使用相对路径，实际工作目录取决于 Tauri 进程启动位置。
建议：使用 tauri::api::path::app_data_dir() 获取标准应用数据目录。
P1 内存与数据库不一致 chatSlice.ts:277-286
addUserMessage 先 set 更新内存，再 invoke 写库。后者失败则内存有数据而数据库无记录。
建议：调整为"写库成功后再改内存"，或引入乐观更新 + 错误回滚机制。
P1 Schema 与 TS 类型不同步 db.rs:23-38 vs types/models.ts
Rust Agent 结构体保留 model_provider、api_key、base_url 等已废弃字段，TS Agent 已使用 providerId、modelId。数据库未同步迁移。
建议：新增 provider_id、model_id 列；将旧字段标记废弃并逐步清理。引入 schema 迁移框架。
P1 无 Schema 版本控制 db.rs
无 PRAGMA user_version 或 _migrations 表。唯一的"迁移"是检测 workflow_id 列是否存在并 ALTER TABLE。
建议：引入 user_version 机制，按版本号顺序执行迁移脚本。创建 _migrations 表记录执行历史。
P1 API Key 明文存储 db.rs, modelSlice.ts
所有服务商 API Key 以明文存储在 SQLite 和 JSON 中，可被任意进程读取。localStorage fallback 进一步扩大了攻击面。
建议：使用 Tauri Stronghold 或系统钥匙串加密存储；禁用 localStorage 回退。优先级：P1（"暂时没那么紧急"）。
P1 无备份与恢复机制 全局缺失
无任何备份机制。simperstudio.db 损坏即永久丢失。无导出/导入 API。
建议：启动/退出时执行 VACUUM INTO 'backup/simperstudio_YYYYMMDD.db'。暴露 export_data() 和 import_data() 命令。
P2 单连接串行瓶颈 db.rs:7-9
Mutex<Connection> 导致所有 DB 操作串行，多 Agent 并发时阻塞。
建议：使用 r2d2_sqlite 连接池管理多连接，允许并行读取。

👀 代码审查师审计

审计范围：正确性、安全性、可维护性、性能、测试覆盖、代码模式、错误处理、边界条件

阻塞项 (Blocking Issues)

P0 withTimeout 存在 timeoutId 未定义风险 helpers.ts:3-9
timeoutId 声明后若在 setTimeout 回调执行前被 clearTimeout，存在 undefined 风险。若 ms <= 0，赋值可能未完成。
建议：将类型改为 NodeJS.Timeout | undefined，finally 中检查 if (timeoutId !== undefined) 再清除。
P0 evaluateExpression 使用 new AsyncFunction + with(payload) helpers.ts:31-39
与架构师审计交叉确认：工作流条件表达式直接执行用户输入字符串。with(payload) 无法阻止原型链污染或全局访问。同步版本 evaluateExpressionSync 同样危险。
建议：使用 jexl/jsonata 替代；若必须保留 JS，用 Web Worker 隔离执行环境。
P0 codeExecutor 直接执行用户 JavaScript 代码 codeExecutor.ts:4-13
Code 节点使用 new AsyncFunction 执行用户代码，10秒超时无法阻止同步恶意操作（如 while(true){} 阻塞事件循环）。
建议：使用 Web Worker 隔离执行；或改用受限 DSL 替代原生 JS；至少静态分析禁止 eval、Function、import 等关键字。
P0 httpExecutor URL 协议与格式风险 httpExecutor.ts:4-34
URL 经模板替换后直接 fetch()，无协议校验（允许 file://）。headers 通过 JSON.parse 解析，非法 JSON 静默失败。body 被 JSON.stringify 包裹后作为字符串发送，Content-Type 可能不匹配。
注：HTTP 节点可访问内网地址（localhost/169.254.x.x）为产品设计要求，不作为缺陷处理。
建议：添加协议白名单 ['http:', 'https:']；headers 解析失败时显式报错；校验 Content-Type 与 body 格式一致性。
P0 apiKey || 'sk-custom' 两处逻辑不一致 api.ts:80
resolveAgentModelConfig 已检查 !provider.apiKey 并抛出错误，但 fetchFromProvider 仍保留 || 'sk-custom' fallback，产生无意义请求。
建议：移除 || 'sk-custom' fallback，统一由 resolveAgentModelConfig 校验。
P0 activeProviderId fallback 与默认值矛盾 agentProviderRouter.ts:30, modelSlice.ts:93
modelSlice 默认 activeProviderId: 'siliconflow-default'，但所有 provider isEnabled 默认为 false。导致 resolveAgentModelConfig 的 !provider.isEnabled 校验必失败。
建议：将 siliconflow-default 的 isEnabled 设为 true，或修改 autoSelectActiveProvider 逻辑与默认值一致。

建议项 (Suggestions)

P1 chatSlice.ts 多处空 catch {} 静默吞错 chatSlice.ts (多处)
Tauri 后端调用失败被完全静默处理，如 try { await invoke('add_chat_session', ...) } catch { }。用户无法感知数据保存失败。
建议：至少记录 console.error，或向用户显示 toast 通知。
P1 sendMessageToAgents 使用 Promise.all 错误处理不完善 chatSlice.ts:414-422
任一 agent 调用失败，Promise.all 立即 reject，其他 agent 响应被丢弃。错误未被外层 catch。
建议：使用 Promise.allSettled 替代，分别处理每个 agent 的成功/失败状态。
